# 智能信贷审批 Agent 项目完善指导书

## 1. 指导书目的

这份文档的目标不是继续补充“项目介绍”，而是告诉你接下来应该怎么自己把项目一步一步写完整。

你现在已经有了：

- 项目计划书
- API 文档
- 数据库建表 SQL
- 技术注意点
- Java 项目骨架

下一步最重要的事情，不是继续扩展概念，而是按顺序把项目真正落地。

这个项目建议按照：

`先跑通主流程 -> 再补数据库 -> 再补 Agent -> 再补工具 -> 再补 RAG -> 再补工程能力`

的顺序推进。

不要一开始就同时写：

- 数据库
- 文件上传
- OCR
- LangChain4j
- RAG
- 状态机
- MQ
- 脱敏
- 监控

这样非常容易失控。

---

## 2. 总体开发顺序建议

推荐拆成 6 个阶段：

1. 第一阶段：把基础 CRUD 和主审批流程跑通
2. 第二阶段：把数据库和持久化补完整
3. 第三阶段：把 Agent 分工和工具调用落下来
4. 第四阶段：把 RAG 知识库接进审批决策
5. 第五阶段：把状态机、审计、异常处理补齐
6. 第六阶段：做优化、监控、脱敏和面试包装

你后续写项目时，必须遵守一个原则：

`每完成一个阶段，就确保项目是能运行、能演示、能讲清楚的。`

---

## 3. 第一阶段：先把主流程跑通

### 3.1 这一阶段的目标

先不要追求“智能”，而要追求“流程通”。

这一阶段的目标是：

- 用户可以提交贷款申请
- 后端可以保存申请单
- 可以查询申请状态
- 可以模拟走完一次审批流程
- 可以得到一个审批结果

### 3.2 这一阶段该写什么

优先写这些内容：

1. `LoanApplicationController`
- 创建申请
- 查询申请详情
- 查询申请状态

2. `LoanApplicationService`
- 创建申请单
- 保存状态
- 返回结构化响应

3. `LoanApplication` 实体
- 先把核心字段补全
- 至少要有申请人信息、金额、期数、状态、创建时间

4. `AgentOrchestrationService`
- 不需要真正调用大模型
- 先用固定逻辑模拟完整流程

5. 审批结果返回
- 固定返回通过/拒绝/人工复核中的一种

### 3.3 这一阶段的完成标准

做到下面几点就算完成：

- Postman 能成功调创建申请接口
- 能查到申请记录
- 能看到申请状态变化
- 能返回一份固定审批结果

### 3.4 这一阶段不要做什么

- 不要急着接 LangChain4j
- 不要急着接 MySQL
- 不要急着做文件上传
- 不要急着做 Redis
- 不要急着做 MQ

因为第一阶段的核心只有一句话：

`把审批主流程路径打通。`

---

## 4. 第二阶段：把数据库接起来

### 4.1 目标

把当前项目从“内存演示版”升级成“真实后端版”。

### 4.2 推荐你先接什么表

第一批先接最核心的 4 张：

1. `loan_application`
2. `approval_decision`
3. `manual_review_ticket`
4. `state_transition_log`

先不要一上来把十几张表全接完。

### 4.3 推荐写法

如果你是 Java 后端求职方向，建议优先：

- `Spring Boot + MyBatis`

这样更贴近大多数业务系统。

目录建议继续补：

- `mapper`
- `mapper/xml`
- `service/impl`

### 4.4 这一阶段具体怎么写

1. 先建库建表
2. 配置 `application.yml` 里的数据库连接
3. 写 `LoanApplicationMapper`
4. 写 `ApprovalDecisionMapper`
5. 把当前内存存储替换成数据库读写
6. 每次状态流转时写一条状态日志

### 4.5 完成标准

- 创建申请会真正写入数据库
- 查询接口能从数据库读数据
- 审批结果能落库
- 状态流转可查

### 4.6 常见坑

- 实体字段和 SQL 字段对不上
- 枚举直接写数据库导致类型不统一
- 创建申请成功但状态日志没写
- 人工复核后主表状态没有同步更新

---

## 5. 第三阶段：完善 Agent 和工具调用

### 5.1 目标

把现在的“占位 Agent”升级成“有职责、有输入输出”的真实模块。

### 5.2 推荐顺序

不要一起写四五个 Agent，建议按下面顺序：

1. 信息采集 Agent
2. 反欺诈风控 Agent
3. 偿债能力 Agent
4. 合规决策 Agent

### 5.3 每个 Agent 怎么写

每个 Agent 至少要包含：

- 一个清晰的方法入口
- 一个输入 DTO
- 一个输出 DTO
- 一个工具调用层
- 一个日志记录点

推荐结构：

- `agent`
- `tool`
- `dto/agent`

### 5.4 先不用真接第三方接口

你完全可以先做“本地模拟工具”：

- OCR 工具先写死返回解析结果
- 黑名单工具先根据身份证尾号决定是否命中
- 收入分析工具先根据金额简单计算 DTI

这样做的意义是：

先把工具调用链路跑通，再逐步替换成真实实现。

### 5.5 完成标准

- 每个 Agent 都有输入输出对象
- 编排层能按顺序调用 Agent
- 每个 Agent 的结果能打印或落日志
- 决策 Agent 能整合前几个 Agent 的结果

### 5.6 常见坑

- Agent 之间直接传 entity，耦合太重
- 所有 Agent 返回 String，自由文本难处理
- 工具层和业务层写在一起，后续不好替换

---

## 6. 第四阶段：接入 LangChain4j 和 RAG

### 6.1 目标

把项目从普通审批系统升级成真正的 Agent 项目。

### 6.2 接入顺序

建议先接 LangChain4j，再接 RAG。

顺序不要反。

原因是：

- 先把模型调用跑通
- 再让模型带着检索结果做决策

### 6.3 先做哪一类能力

建议第一批只让 LLM 做两件事：

1. 风险摘要生成
2. 决策解释生成

不要第一步就让大模型直接输出最终审批结论。

### 6.4 RAG 怎么落地

第一版先做轻量化：

- 准备几份信贷政策文本
- 做条款切分
- 存入向量库或简单检索结构
- 查询时按关键词和向量混合召回

如果你当前时间有限，甚至可以先这样做：

- 不接完整向量数据库
- 先做本地政策文本检索 + 关键词匹配

等主流程稳定后再替换成：

- PGVector
- Elasticsearch
- Milvus

### 6.5 Prompt 设计建议

每个 Agent 的 Prompt 都要限制边界：

- 只能基于输入字段和检索条款输出
- 缺信息时返回固定标记
- 输出固定 JSON
- 禁止自由发挥

### 6.6 完成标准

- 项目能够成功调用一次大模型
- Agent 能基于工具结果输出结构化摘要
- 决策 Agent 能基于检索条款生成解释文本

### 6.7 常见坑

- 模型输出格式不稳定
- Prompt 写太长，后续难维护
- RAG 检索内容太多，导致结果飘
- 检索不到条款时没有兜底动作

---

## 7. 第五阶段：补状态机、日志和异常处理

### 7.1 目标

这一阶段要把项目从“能跑”升级到“像业务系统”。

### 7.2 重点补什么

1. 状态机
- 申请提交
- 材料待补充
- 风控中
- 决策中
- 人工复核
- 审批完成

2. 状态流转日志
- 谁触发了状态变化
- 从哪个状态变到哪个状态
- 为什么变化

3. 工具调用日志
- 调了什么工具
- 请求参数摘要
- 响应结果摘要
- 是否成功

4. 异常处理
- 工具失败
- 参数错误
- 状态不合法
- 第三方接口超时

### 7.3 具体建议

- 先不用复杂状态机框架也可以
- 第一版可用“状态枚举 + 状态校验方法 + 状态日志表”
- 后续再升级成 Spring StateMachine

### 7.4 完成标准

- 每一步状态变化都能查
- 工具异常不会直接把服务打挂
- 人工复核流可以单独进入和退出

---

## 8. 第六阶段：做工程化增强

### 8.1 这一阶段做什么

当主流程已经稳定后，再补这些能力：

- Redis 缓存
- MQ 异步任务
- 重试与熔断
- 敏感信息脱敏
- 监控埋点
- 审批报告导出

### 8.2 推荐顺序

1. 先做审计日志
2. 再做脱敏
3. 再做 Redis
4. 再做 MQ
5. 最后做监控和报表

### 8.3 为什么这样排

因为：

- 日志和脱敏影响最基础的安全性
- 缓存和 MQ 是优化项
- 监控是锦上添花，但也很重要

---

## 9. 你实际编码时的推荐顺序

下面是一套很适合你自己按天推进的编码顺序。

### 第 1 天

- 补全实体类字段
- 完成创建申请接口
- 完成查询申请接口

### 第 2 天

- 接 MySQL
- 写 mapper
- 把申请落库

### 第 3 天

- 写审批状态更新逻辑
- 写状态流转日志

### 第 4 天

- 写信息采集 Agent
- 写本地模拟 OCR 工具

### 第 5 天

- 写反欺诈 Agent
- 写黑名单和规则模拟工具

### 第 6 天

- 写偿债能力 Agent
- 写 DTI 和月供计算工具

### 第 7 天

- 写合规决策 Agent
- 输出固定审批结论和解释文本

### 第 8 天

- 接 LangChain4j
- 让 LLM 生成风险摘要

### 第 9 天

- 接简单 RAG
- 基于政策条款生成解释

### 第 10 天

- 写人工复核接口
- 写审批报告生成逻辑

这套顺序的好处是：

- 每天都有明确产出
- 每一步都能演示
- 不容易陷入“写了很多但跑不起来”

## 10. 你每一阶段都要问自己的问题

### 第一类问题：主流程是否真的通了

- 能不能创建申请
- 能不能查询状态
- 能不能拿到审批结果

### 第二类问题：这一层是不是可替换

- 现在的内存存储能不能替换成数据库
- 现在的模拟工具能不能替换成真实工具
- 现在的固定规则能不能替换成规则引擎

### 第三类问题：是否方便面试表达

- 这一块我能不能讲清楚
- 为什么要这么设计
- 如果面试官追问风险和合规，我能不能接得住

---

## 11. 你后面自己写时的几个硬性原则

### 原则 1：不要一口气写太多层

每次只推进一小段：

- 先接口
- 再 service
- 再 mapper
- 再 agent

### 原则 2：每个模块都要能单独解释

比如：

- 信息采集 Agent 是干什么的
- 风控 Agent 和决策 Agent 的边界是什么
- 为什么要单独有人工复核

### 原则 3：先用最简单方案跑通，再升级

例如：

- 先内存存储，再数据库
- 先本地模拟工具，再真实接口
- 先关键词检索，再向量检索
- 先同步流程，再异步化

### 原则 4：结构化优先

无论是 Agent 输出、日志记录还是审批结果，尽量结构化，不要一开始全写成自然语言。

---

## 12. 后期怎么包装成一个好项目

当项目基本完成后，你还要再做三件事：

### 1. 补项目亮点

比如：

- 多 Agent 分工
- RAG 约束审批
- 规则引擎 + LLM 三层决策
- 审计日志与人工接管

### 2. 补量化指标

比如：

- 平均审批耗时
- 自动审批覆盖率
- 人工复核占比
- 工具调用成功率

### 3. 补演示路径

你要能完整演示：

- 提交申请
- 查看状态变化
- 输出审批结论
- 查看报告或日志

---

## 13. 最终建议

你后面自己完善这个项目时，不要追求“代码一下子写完”，而要追求：

- 每一步都能运行
- 每一步都能解释
- 每一步都能继续往下扩

最稳的路线就是：

`先跑通 -> 再结构化 -> 再智能化 -> 再工程化 -> 再包装面试`

如果你按这条路线写，这个项目会非常适合你继续自己完成，也非常适合后续写进简历和准备面试。
