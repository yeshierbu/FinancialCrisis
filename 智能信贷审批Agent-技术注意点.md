# 智能信贷审批 Agent 技术注意点

## 1. 文档说明

本文档总结“智能信贷审批 Agent”项目在设计和实现过程中需要重点关注的技术注意点，适用于开发落地、代码评审、系统设计答辩和面试准备。

这个项目不是普通的聊天式 Demo，而是一个带有审批流程、风控逻辑、知识检索、工具调用、审计留痕和人工兜底能力的金融类系统，因此在实现时要特别注意稳定性、可解释性、合规性和工程边界。

---

## 2. 不要把大模型当作最终审批器

这是整个项目最核心的注意点。

在金融场景中，大模型可以参与：

- 非结构化材料理解
- 风险摘要生成
- 政策条款整合
- 决策解释输出

但不能直接作为最终审批结论的唯一依据。

技术上建议采用三层机制：

1. 规则引擎负责硬性准入和拒贷条件判断
2. RAG 负责提供政策依据和案例支撑
3. LLM 负责解释、总结和结构化输出

注意事项：

- 禁止让模型在没有命中规则和条款的情况下自由生成审批结论
- 高风险样本必须由规则或人工复核兜底
- 所有拒贷理由都要有原因码和条款引用

---

## 3. 多 Agent 设计不要只停留在“拆几个类”

很多项目会把多个 Service 命名成 Agent，但实际上仍然是串行代码，缺少真正的职责边界。

本项目里多 Agent 的关键不在于名字，而在于：

- 每个 Agent 有独立职责
- 每个 Agent 有独立输入输出结构
- 每个 Agent 的失败可以被单独处理
- Orchestrator 统一编排全局流程

注意事项：

- 不要让所有 Agent 共用一个超大 Prompt
- 不要让所有 Agent 输出自由文本，尽量统一输出 JSON
- 每个 Agent 的职责要尽量单一，避免提示词膨胀
- Agent 之间传递的是结构化结果，而不是散乱自然语言

---

## 4. RAG 一定要做“约束式检索”，不要只做知识问答

这个项目里的 RAG 不是为了聊天，而是为了限制决策边界。

注意事项：

- 文档要按条款级切分，而不是粗粒度大段切分
- 每个切片要带上 `policy_id`、`clause_id`、生效日期、适用产品等元数据
- 混合检索比纯向量检索更稳，建议 `BM25 + 向量检索 + Rerank`
- 检索不到有效条款时，应返回“建议人工复核”，而不是让模型补全

常见坑：

- 只存纯文本，不存元数据，后续无法做条款引用
- 一次性召回太多内容，导致 Prompt 太长、结论变飘
- 政策文档版本不管理，导致过期规则参与审批

---

## 5. Tool Calling 要可控，不要让 Agent 无限调用

LangChain4j 的 Tool Calling 很适合这个项目，但必须加边界。

注意事项：

- 每个工具都要有超时设置
- 每个工具都要定义重试次数
- 每个工具都要有幂等保障
- 工具失败后要有明确兜底动作
- Agent 的循环次数必须限制，避免死循环调用

建议：

- 工具执行统一加日志
- 工具结果统一做结构化封装
- 第三方接口错误不要直接抛给前端，应转换成业务可理解状态

常见坑：

- 把第三方接口返回原样暴露给 Agent，导致结果格式不可控
- 工具入参与出参不固定，后续调试困难
- 没有失败补偿策略，导致一个接口超时拖垮整个审批流

---

## 6. 状态机是必要的，不要只靠一个 status 字段硬写 if-else

因为这个项目是长流程审批系统，不是单次请求响应，所以一定要有显式状态流转管理。

建议至少管理这些状态：

- SUBMITTED
- MATERIAL_PENDING
- OCR_PARSING
- EXTERNAL_VERIFYING
- RISK_ANALYZING
- DECISIONING
- MANUAL_REVIEW
- APPROVED
- REJECTED
- ARCHIVED

注意事项：

- 所有接口都要做状态校验
- 状态迁移要记录日志
- 状态迁移失败要能回滚或补偿
- 人工复核必须是独立状态，不能混在自动审批中

常见坑：

- 前端重复提交导致状态错乱
- 并发更新把状态覆盖掉
- 工具超时后流程卡死，没有补偿扫描

---

## 7. 审批结果一定要可解释、可回放

金融项目最怕“结果出来了，但没人知道为什么”。

注意事项：

- 每次审批都要记录命中的风险标签
- 每次审批都要记录命中的规则编码
- 每次审批都要记录引用的政策条款
- 每次审批都要记录 Agent 输出摘要和工具调用摘要

建议存储：

- `agent_task_log`
- `tool_call_log`
- `llm_trace_log`
- `policy_hit_record`
- `state_transition_log`

如果面试官问“怎么做审计留痕”，这一部分就是最重要的回答点。

---

## 8. 敏感信息不能直接进日志和 Prompt

这是非常容易被忽略的问题。

项目中涉及：

- 身份证号
- 手机号
- 银行卡号
- 家庭住址
- 征信信息
- 联系人信息

注意事项：

- API 请求日志要脱敏
- LLM Prompt 入参要脱敏
- 工具调用日志要脱敏
- 管理端列表页要脱敏
- 审批报告如果给非授权人员查看，也要做展示级脱敏

实现建议：

- 优先使用 `AOP + 注解 + Jackson Serializer`
- 如果想做项目亮点，可以再补充 `Java Agent + ByteBuddy`

但从落地角度看，先把简单稳定的脱敏机制做好，比追求高级方案更重要。

---

## 9. 外部接口一定会失败，必须提前设计降级方案

本项目会依赖很多外部能力：

- OCR
- 征信接口
- 黑名单接口
- 手机实名接口
- 设备风险接口

注意事项：

- 给所有第三方接口设置超时
- 设置有限次重试，不要无限重试
- 必须有 fallback 方案
- fallback 不能等于“自动通过”

推荐策略：

- OCR 失败：提示用户补传或转人工
- 征信失败：尝试缓存数据，并标记非实时
- 黑名单接口失败：强制人工复核
- RAG 未命中政策：转人工，不做自动拒贷

推荐技术：

- `Resilience4j`
- `RabbitMQ / Kafka`
- 定时补偿任务

---

## 10. 不要忽略并发和幂等问题

审批类项目经常会遇到重复提交和重复处理问题。

需要重点控制的场景：

- 用户重复点击“提交申请”
- 用户反复上传同一份材料
- 消息重复消费
- 工具重试导致重复写库
- 人工复核接口重复点击

注意事项：

- 创建申请要做幂等校验
- 上传材料要有文件 hash 或业务唯一标识
- 工具执行日志要带 traceId
- 人工审核通过/拒绝要基于状态校验和乐观锁

常见技术手段：

- Redis 幂等键
- 数据库唯一索引
- 乐观锁版本号
- MQ 消息去重

---

## 11. 表结构设计时要优先考虑“查审计”而不是“查展示”

很多人设计表只想着页面怎么展示，但这个项目真正重要的是后续能不能回放审批过程。

注意事项：

- 核心结果表要结构化
- JSON 字段可以有，但不要所有信息都塞 JSON
- 风险标签和条款引用要保留单独日志或明细
- 时间字段要完整，便于还原处理顺序

建议：

- 主流程表存最终状态
- 结果表存结构化分析
- 审计表存过程日志

这样后续查问题、做指标、做回放都会轻松很多。

---

## 12. Prompt 设计要追求稳定，不要追求“写得像人”

这是 Agent 项目里很重要的一点。

在审批场景下，Prompt 的目标不是优美表达，而是稳定输出。

注意事项：

- 尽量限定输出字段
- 尽量明确禁止自由发挥
- 尽量要求基于工具结果和检索条款作答
- 输出固定 JSON，必要时做 Schema 校验

建议：

- 不同 Agent 使用不同 Prompt
- 把公共约束抽成模板
- 增加异常场景提示词，例如“若信息不足则返回 NEED_MANUAL_REVIEW”

常见坑：

- 一个 Prompt 想解决所有场景
- 输出既要自然语言又要结构化 JSON，导致格式漂移
- 没有 Schema 校验，后端解析频繁失败

---

## 13. 报告生成要区分“内部审计版”和“外部展示版”

审批报告看起来只是导出文件，但实际上很容易涉及权限问题。

建议至少区分两类报告：

1. 内部审计版
- 包含完整工具调用摘要
- 包含命中规则和政策依据
- 包含风险标签和中间指标

2. 用户展示版
- 只展示最终审批结果
- 拒贷时只展示合规允许披露的标准原因
- 不展示内部风控细节

注意事项：

- 不同版本报告字段权限不同
- 报告生成也要写日志
- 报告文件地址要受权限控制

---

## 14. 指标监控不要只监控接口 QPS

这个项目的监控不能只停留在传统接口维度。

建议监控三类指标：

1. 系统指标
- Agent 平均耗时
- 工具调用成功率
- 外部接口超时率
- OCR 成功率
- LLM token 消耗

2. 业务指标
- 自动审批通过率
- 人工复核占比
- 补件率
- 拒贷率
- 风险等级分布

3. 模型治理指标
- 条款引用准确率
- 人工纠正率
- 决策一致率
- JSON 解析失败率

如果你后面想做项目优化，这些指标就是最好的量化抓手。

---

## 15. 第一版落地要控制复杂度

这个项目很容易越做越大，所以一定要分阶段。

建议第一版只做：

- 申请提交
- 文件上传
- OCR 解析
- 简单规则审批
- 一个基础 Agent 编排流程
- 一版审批报告

第二版再加：

- RAG
- 人工复核工单
- 风控标签体系
- 审计日志
- 监控埋点

第三版再加：

- 异步化
- 熔断重试
- 缓存优化
- 脱敏增强
- 多模型切换

注意事项：

- 不要第一版就追求全部技术栈一起上
- 优先保证主流程能走通
- 优先保证结果可解释和可演示

---

## 16. 面试表达上的技术重点

如果后面要用于面试，建议重点强调以下几点：

1. 为什么采用多 Agent，而不是单 Agent
2. 为什么不能让大模型直接做最终审批
3. 如何通过 RAG 和规则引擎降低幻觉风险
4. 如何保证审批结果可解释、可回放、可审计
5. 如何处理外部工具失败、流程中断和人工接管

这些问题回答得好，项目说服力会明显提高。

---

## 17. 总结

这个项目最重要的不是“用了大模型”，而是如何把大模型合理放进一个金融审批系统里。

真正能拉开差距的点有四个：

- 多 Agent 职责划分是否合理
- 规则引擎、RAG、LLM 的边界是否清晰
- 状态机、日志、脱敏、降级是否完整
- 整个审批链路是否可解释、可追溯、可人工接管

只要把这些技术注意点落实好，这个项目就会比普通 AI Demo 更像一个真实可落地的金融科技项目。
